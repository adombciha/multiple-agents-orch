# Gemini Skill: test-proposal-generator

## Description

Generate structured, automation-ready testing scenarios and test proposals from a clearly defined user-provided test scope.

This skill plans tests only. It does not generate test code.

## Instructions for Gemini

1. **Check Scope First**: Before generating a proposal, confirm that the user-provided scope defines:
   - the system, module, function, algorithm, or data flow under test
   - the test objective
   - included behavior
   - excluded behavior
   - high-level success and failure criteria
   - known risks, regression concerns, or release gate requirements
   - whether the scope requires full validation, release evidence, or biological/clinical correctness planning
2. **Apply Expected Behavior Priority**: Use `SPEC > CODE > UI behavior reasonableness`. If SPEC exists, test against SPEC; CODE only verifies implementation conformance. If no SPEC but CODE exists, generate characterization/regression tests only. If neither SPEC nor CODE exists but UI behavior is clear, generate UI behavior tests only and state that algorithm/code correctness is out of scope.
3. **Handle Function Name Input**: When the user provides a function name, perform read-only code discovery first. Resolve the definition, file/module/signature, callers, public entry points, inputs, outputs, direct callees, helper functions, state changes, side effects, and error paths. If the symbol is ambiguous, ask for file/module or exact signature before generating the proposal.
4. **Handle White-Box Scope**: For white-box requests, inspect every visible function in the requested file, class, module, or reachable flow. For each function, record purpose, inputs, outputs, branch conditions, loops, numerical logic, state mutation, side effects, error paths, boundary cases, and testability. Classify each as `In Primary Flow`, `Direct Dependency`, `Utility / Low Risk`, or `Out of Requested Flow`.
5. **Apply TDD-Oriented Planning**: For spec-based scopes, identify the first failing behavior each generated test should prove. For bug fixes, target the smallest public-boundary behavior that reproduces the bug. For existing code without SPEC or golden outputs, generate characterization/regression scenarios and state that they preserve observed behavior rather than prove correctness.
6. **Ask Blocking Questions When Needed**: If the target, boundary, expected behavior, pass/fail criteria, golden reference source, data ownership, or release gate decision is unclear, output `Scope Clarification Required` and ask blocking questions before generating a proposal.
7. **Handle Non-Blocking Details**: If only naming, priority, folder layout, report column names, optional cases, or default tolerance details are unclear, continue with explicit assumptions and mark affected items as `Pending RD Confirmation`.
8. **Generate QA Proposal**: When the scope is clear enough, produce an `Automated Test Proposal` with:
   - scope summary
   - QA interpretation
   - test objective
   - included and excluded scope
   - system under test
   - code logic boundary
   - function inventory and white-box coverage classification, when applicable
   - UI behavior boundary
   - testing scenarios
   - required inputs and configuration
   - required expected outputs or golden reference
   - test case matrix
   - boundary and abnormal cases
   - regression risks covered
   - comparison and tolerance rules
   - report requirements
   - traceability requirements
   - release gate criteria
   - biological / clinical validation planning
   - RD confirmation points
   - risks and assumptions
   - downstream generation notes
   - the downstream report-generation command using the `scripts/generate_test_report.py` provided alongside this SKILL.md file
9. **Golden Reference Rule**: If expected outputs do not have an approved reference source, state that the proposal can guide automation but cannot be used as release evidence until the golden reference is approved.
10. **Biological / Clinical Validation Rule**: Do not assume biological or clinical correctness is excluded. If the scope is full validation or release evidence, include biological/clinical correctness in validation planning, while clearly stating that demo data is not clinical evidence.
11. **Report Ownership Rule**: Final test reports and RD action reports must be generated by the `scripts/generate_test_report.py` script provided alongside this SKILL.md file, not hand-written by the AI agent. The downstream workflow should rerun tests and this report generator to produce second and later reports after fixes.
12. **Compact Chat Rule**: When writing files is part of the task, save the full proposal to the requested path and keep chat output to the path, scope summary, blocking gaps, RD confirmation count, and downstream next step. Paste the full proposal in chat only when the user explicitly asks for it or no file output is requested.
13. **No Test Code**: If the user asks for test code, generate the proposal first and explain how a downstream automation skill should convert it into automated tests and reports.
14. **Quality Bar**: The proposal must be specific, auditable, traceable, suitable for automation, explicit about pass/fail criteria, strict about scope boundaries, and clear about missing approvals.

## Scope Clarification Required Format

```markdown
# Scope Clarification Required

## Why Proposal Cannot Be Generated Yet

Explain exactly which parts of the scope are insufficient and why that prevents reliable testing scenario design or downstream automation generation.

## Missing Scope Boundaries

| Missing Boundary | Why It Matters |
| ---------------- | -------------- |

## Blocking Questions for RD

| ID | Question | Why It Blocks Proposal |
|---|---|

## Example of Acceptable Scope

Provide an example scope that would be specific enough to proceed.
```

Do not generate a test case matrix, tolerance rules, report requirements, or release gate criteria in this mode.

```markdown
## 23. Downstream Generation Notes

- If no implementation language is specified, recommend Python3 tests by default without blocking proposal generation.
- Use the oracle priority `SPEC > CODE > UI behavior reasonableness` when choosing expected behavior.
- Preserve function inventory and white-box coverage classification in downstream generated tests and reports.
- Include first-failing behavior for TDD-ready cases; mark code-only legacy cases as characterization/regression unless an approved SPEC or golden oracle exists.
- For Qt/C++ software scopes, recommend QtTest when C++ core behavior, Qt GUI, widget workflow, signal/slot, or model/view validation is required.
- Use `pytest` only when the repository already uses pytest or the user explicitly approves it.
- Treat Python3 and QtTest as complementary layers when the scope includes both Qt/C++ behavior and release evidence/report validation.
- Generate final reports using the script bundled with this skill. Find its absolute path (it is located at `scripts/generate_test_report.py` relative to this `SKILL.md` file) and run: `python3 <absolute_path_to_skill_script> --spec <output-dir>/markdown-test-spec.json --results <output-dir>/test-results.json --out-dir <output-dir>`.
- Fixed report outputs are `test-report.json`, `test-report.md`, `rd-action-report.json`, and `rd-action-report.md`.
- After fixes, rerun the tests and `generate_test_report.py`; do not rerun this proposal skill only to produce a second report.
```
